# LOCAL_SETUP_AGENT

Инструкция для AI-агента, который должен подготовить локальный компьютер к работе с этим репозиторием, проверить окружение, по возможности установить всё сам и остановиться только там, где действительно нужен пользователь.

## Назначение

Этот документ нужен для **локального режима**, когда skill'ы запускаются не в Claude Projects, а на компьютере пользователя.

Задача агента:

1. Проверить, что уже установлено.
2. Доустановить всё, что можно поставить автоматически.
3. Аккуратно заменить platform-specific возможности (`project_knowledge_search`, `view`, `present_files`) локальными эквивалентами.
4. Если требуется ручное действие, не гадать, а коротко объяснить пользователю:
   - что именно нужно сделать;
   - зачем это нужно;
   - как проверить, что шаг выполнен.

## Что считать успехом

Локальная машина готова, если:

- доступны `Python` и `pip`;
- установлен пакет `python-pptx`;
- доступны `Node.js` и `npm`;
- установлен npm-пакет `docx`;
- агент умеет упаковывать и распаковывать ZIP локально;
- есть понятная локальная замена для `project_knowledge_search`;
- есть понятная локальная замена для `present_files`;
- есть способ открыть локальные изображения для шагов, где нужен визуальный просмотр;
- пользователь предупреждён о шагах, которые нельзя полностью автоматизировать локально.

## Важное ограничение

Не все зависимости этого репозитория являются "устанавливаемыми программами".

Если инструкции в разных файлах расходятся, агент должен применять такой приоритет:

1. конкретный `SKILL.md` нужного skill'а;
2. локальный скрипт, если он уже существует в репозитории;
3. pipeline-документы;
4. общие файлы вроде `README.md` и `CLAUDE.md`.

Часть skill'ов опирается на возможности хост-платформы:

- `project_knowledge_search`
- `view`
- `present_files`

Их не нужно "устанавливать" как отдельные пакеты. Для локального режима агент должен организовать им замену:

- `project_knowledge_search` -> локальный поиск по рабочим папкам пользователя;
- `view` -> открытие локальных изображений или просмотр через клиент, который умеет показывать картинки;
- `present_files` -> сохранение результатов в выходную папку и явное сообщение пользователю пути к файлу.

## Что агент должен уметь поставить сам

Агент должен пытаться установить автоматически:

- `Python`
- `pip`
- `python-pptx`
- `Node.js`
- `npm`
- npm-пакет `docx`
- вспомогательные локальные папки:
  - `input/`
  - `output/`
  - `knowledge/`
  - `tmp/`

Если на машине Windows доступны пакетные менеджеры, агент может использовать:

- `winget`
- `choco`
- `scoop`

Если пакетный менеджер недоступен, агент не должен зацикливаться. Нужно остановиться и попросить пользователя установить отсутствующий компонент вручную.

## Что агент не должен обещать автоматизировать полностью

Следующие вещи обычно требуют ручного участия пользователя:

- вход в внешние сервисы;
- генерация иллюстраций через `ChatGPT`, `DALL-E` или другой внешний image generator;
- установка проприетарных шрифтов, если у агента нет легального локального файла шрифта;
- любые действия, где ОС просит UAC-подтверждение или вмешивается корпоративная политика безопасности.

## Обязательные проверки

Перед установкой агент должен проверить:

### 1. Базовые команды

- `python --version` или `py --version`
- `pip --version` или `python -m pip --version`
- `node --version`
- `npm --version`

### 2. Python-зависимости

Проверить импорт:

```powershell
python -c "import pptx; print('python-pptx ok')"
```

Если команда падает, установить:

```powershell
python -m pip install python-pptx
```

### 3. Node/npm-зависимости

Проверить пакет `docx`:

```powershell
npm list -g docx --depth=0
```

Если пакет не найден, установить:

```powershell
npm install -g docx
```

### 4. Архивация

На Windows не обязательно требовать внешние `zip` и `unzip`.

Допустимые замены:

- `Compress-Archive`
- `Expand-Archive`
- Python-модуль `zipfile`

Если стандартные PowerShell-команды доступны, этого достаточно.

### 5. Шрифт для иврита

Skill'ы DOCX ориентируются на шрифт `David`.

Агент должен:

1. Проверить, установлен ли `David`.
2. Если шрифт есть, просто отметить это.
3. Если шрифта нет:
   - предупредить пользователя;
   - объяснить, что DOCX всё равно может сгенерироваться, но визуальная проверка может отличаться;
   - предложить пользователю вручную установить `David` или подтвердить использование системного fallback.

Агент не должен скачивать шрифты из сомнительных источников.

## Рекомендуемая локальная структура

Если пользователь не задал свою структуру, агент должен предложить и при согласии создать:

```text
<workspace>/
├── skills/        # этот репозиторий
├── input/         # входные файлы пользователя
├── output/        # готовые результаты
├── knowledge/     # локальная замена Project Knowledge
└── tmp/           # временные файлы
```

Если репозиторий уже открыт как workspace, достаточно создать рядом или внутри рабочей директории папки:

- `input`
- `output`
- `knowledge`
- `tmp`

## Локальные замены platform-specific функций

### `project_knowledge_search`

Локальная стратегия:

1. Искать сначала в `knowledge/`.
2. Затем в `input/`.
3. Затем в рабочей директории пользователя.
4. Использовать быстрый поиск по имени и по содержимому.

Рекомендуемые локальные команды:

```powershell
Get-ChildItem -Recurse -File
rg "нужный_фрагмент" knowledge input
rg --files knowledge input
```

### `view`

Локальная стратегия:

- если клиент агента умеет показывать локальные изображения, использовать путь к файлу;
- иначе открыть изображение стандартным приложением ОС;
- если ни один вариант недоступен, попросить пользователя вручную открыть файл и подтвердить, что изображение видно.

### `present_files`

Локальная стратегия:

- сохранять результат в `output/`;
- сообщать пользователю абсолютный путь;
- кратко писать, что именно создано.

## Порядок действий агента

### Шаг 1. Определить ОС и доступные пакетные менеджеры

На Windows проверить по порядку:

- `winget`
- `choco`
- `scoop`

Если ничего нет, не пытаться придумывать обходной путь через случайные загрузки из интернета. Вместо этого честно запросить ручную установку нужного runtime.

### Шаг 2. Проверить Python

Если `Python` отсутствует:

- при наличии `winget` можно установить автоматически;
- рекомендуемая команда:

```powershell
winget install --id Python.Python.3 -e
```

- после установки ещё раз проверить `python --version`;
- затем поставить `python-pptx`.

### Шаг 3. Проверить Node.js

Если `Node.js` отсутствует:

- при наличии `winget` можно установить автоматически;
- рекомендуемая команда:

```powershell
winget install --id OpenJS.NodeJS.LTS -e
```

- после установки ещё раз проверить `node --version` и `npm --version`;
- затем поставить `docx`.

### Шаг 4. Проверить архивацию

Если работают `Compress-Archive` и `Expand-Archive`, этого достаточно.

Не требовать внешний `zip.exe`, если встроенные средства уже решают задачу.

### Шаг 5. Создать рабочие папки

Если папки `input/`, `output/`, `knowledge/`, `tmp/` отсутствуют, создать их.

### Шаг 6. Проверить локальные сценарии репозитория

Минимальный smoke test:

1. Проверить, что существует скрипт [extract_pptx.py](C:/git/hebrew-learning-ai-skills/homework/hebrew-lesson-extract/scripts/extract_pptx.py).
2. Проверить, что `python` умеет импортировать `pptx`.
3. Проверить, что `npm list -g docx --depth=0` возвращает пакет.
4. Сформировать короткий отчёт:
   - что уже было установлено;
   - что агент установил сам;
   - что осталось сделать пользователю.

## Когда агент должен остановиться и попросить пользователя

Остановиться и запросить ручной шаг нужно, если:

- нет `winget/choco/scoop`, а нужный runtime отсутствует;
- установка требует UAC или admin rights, которых у агента нет;
- нужен шрифт `David`, но его негде безопасно взять;
- нужен логин во внешний сервис;
- для визуального шага нужно открыть изображение, а локальный клиент этого не умеет.

В таких случаях агент должен задавать не открытый вопрос, а короткую инструкцию вида:

1. Установите `Python 3.x`.
2. Убедитесь, что команда `python --version` работает в новом терминале.
3. Сообщите мне, когда это готово, и я продолжу остальные шаги автоматически.

## Как агент должен формулировать итог

После проверки и установки агент должен выдать итог в трёх списках:

### Установлено

- что уже было на машине;
- что агент установил сам.

### Требует подтверждения

- что пользователь должен сделать руками;
- почему без этого шаг не завершён полностью.

### Локальные ограничения

- какие skill'ы по-прежнему завязаны на внешний сервис;
- какие platform-specific функции были заменены локальными соглашениями.

## Минимальный целевой набор для этого репозитория

Обязательно:

- `Python`
- `pip`
- `python-pptx`
- `Node.js`
- `npm`
- npm-пакет `docx`
- PowerShell с `Compress-Archive` / `Expand-Archive` или эквивалент

Желательно:

- шрифт `David`
- приложение для открытия `.docx`
- приложение для просмотра изображений

Ручной внешний контур:

- `ChatGPT` / `DALL-E` / другой image generator для этапа иллюстраций

## Практическое правило

Если что-то можно безопасно проверить или установить локально, агент должен сделать это сам.

Если шаг нельзя завершить надёжно без участия пользователя, агент должен:

1. остановиться;
2. назвать ровно один блокер;
3. дать короткую инструкцию;
4. после подтверждения пользователя продолжить автоматически.
